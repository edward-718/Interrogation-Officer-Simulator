# 《审讯室》技术迭代建议书

> 分析日期：2026-07-07
> 分析范围：index.html / app.js / engine.js / data/cases.js / styles.css
> 约束条件：纯前端单页应用，无后端、无数据库、无LLM依赖

---

## 一、当前架构速览

| 模块 | 现状 | 能力边界 |
|------|------|----------|
| **对话引擎** (`engine.js`) | 线性遍历 `responseRules`，`tone` 精确匹配 + `matchKeywords` 子串包含，首个命中即返回 | 无上下文记忆、无优先级、无模糊匹配、无否定检测 |
| **状态模型** (`app.js`) | 单维 `mood`（抗拒度 0-100），分 high/mid/low 三档 | 无法表达"紧张但配合""冷静但抗拒"等复合心理状态 |
| **数据层** (`cases.js`) | 案件 → 阶段 → 规则数组，每阶段独立目标/情绪/轮次 | 阶段间无状态传递，规则维护成本高（同义词需逐条重复） |
| **渲染层** | SVG 程序化头像（seed + personality + moodTier）、CSS 动画、打字机效果 | 无语音输出、无音效、无立绘微表情过渡 |
| **持久化** | `localStorage` 仅保存阶段完成数和最后状态 | 不保存对话历史、不支持审讯复盘 |

---

## 二、核心问题：无LLM约束下如何还原创意方案的沉浸感

创意方案中的四大技术需求与Demo级替代方案对照：

| 创意需求 | Demo级替代方案 | 沉浸感还原度 | 实现成本 |
|----------|----------------|--------------|----------|
| AI对话引擎（LLM角色提示词） | **增强型规则引擎**：同义词扩展 + 正则否定检测 + 对话历史感知 + 情绪门控分支 | 约 60%-70% | 低 |
| ASR + TTS | ASR已有（Web Speech API）；**TTS可用 `speechSynthesis`** 为嫌疑人配音 | 约 80% | 极低 |
| 心理状态系统（紧张/防御/信任三维） | **前端三维隐藏模型 + UI微表达**：用立绘抖动、呼吸频率、色温变化间接传达 | 约 50%（无法像LLM一样自然流露，但UI反馈可补足） | 中 |
| 剧情状态机 | **扩展阶段数据结构**：增加 `unlockedConditions` 门控 + 跨阶段 `carryOverState` | 约 70% | 低 |

**核心策略**：用"数据密度 + 感知反馈"弥补"智能深度"的不足。嫌疑人不需要真正理解玩家，只需要在足够多的触发条件下给出看似有逻辑的回应；同时用音效、语音、微动画让玩家"感觉"对面是一个有情绪的人。

## 三、关键词匹配引擎升级方案评估

### 3.1 当前引擎的致命弱点

1. **无否定检测**：玩家输入"我不是主谋"会被 `includes('主谋')` 命中，逻辑上完全相反。
2. **无同义词扩展**："谋划""组织""策划"需在每条规则中手动罗列。
3. **无优先级**：特殊规则和普通规则混排，可能先命中一条通用回复，错过更精准的反应。
4. **无历史感知**：同一句话在第1轮和第5轮说，嫌疑人反应完全相同。

### 3.2 各升级方案评估

| 方案 | 实现复杂度 | Demo收益 | 推荐度 | 说明 |
|------|------------|----------|--------|------|
| **正则否定检测** | 低 | 高 | ★★★★★ | 在 `matchRule` 前增加一层否定词过滤，如 `/(不|没|无|别).{0,3}主谋/` 触发降级回复。代码改动极小，逻辑正确性提升最大。 |
| **同义词扩展表** | 低 | 高 | ★★★★★ | 在 rule 中增加 `matchKeywords` 的自动扩展，或全局维护 `SYNONYM_MAP`。如 `{'主谋': ['主谋','策划','组织者','谋划']}`。 |
| **规则优先级权重** | 中 | 高 | ★★★★☆ | 给 rule 增加 `priority` 字段，匹配后按优先级排序取最高，而非首个命中。可支撑"通用回复→精准回复→情绪门控回复"的层级。 |
| **对话历史感知** | 中 | 高 | ★★★★☆ | 将 `dialogueHistory` 传入 `generateReply`，rule 增加 `requiresUnlocked` / `requiresMoodBelow` 等门控条件。沉浸感质变。 |
| **模糊匹配（编辑距离）** | 中 | 中 | ★★☆☆☆ | 中文模糊匹配需要分词支撑，纯前端收益不如"扩展同义词表"直接。错别字场景有限，不推荐优先做。 |
| **简单意图分类器** | 高 | 中 | ★☆☆☆☆ | 前端手写朴素贝叶斯或关键词加权分类，维护成本高，Demo阶段ROI低。 |

### 3.3 推荐引擎升级路径（按顺序）

```
Step 1: 正则否定检测（当天可完成）
Step 2: 同义词扩展表（当天可完成）
Step 3: 规则优先级 + 历史门控（2-3天）
Step 4: 三维心理模型映射到回复选择（3-5天）
```

## 四、Demo级快速落地增强点

### 4.1 必须做（核心体验，1-3天可完成）

| 增强点 | 具体实现 | 预期效果 |
|--------|----------|----------|
| **嫌疑人语音（TTS）** | 使用 Web Speech API `speechSynthesis`，根据 personality 选择不同语音参数：傲慢型语速慢、音调低；紧张型语速快、音调颤。每条 reply 生成后自动播报。 | 沉浸感提升最显著。玩家从"读文字"变为"听对方说话"。 |
| **否定检测引擎** | `engine.js` 的 `matchRule` 增加 `negationPatterns` 参数，若检测到否定前缀则跳过命中，进入 `negatedReply` 或 fallback。 | 解决当前最突出的逻辑Bug（"我不是主谋"被误判）。 |
| **同义词扩展** | 全局 `SYNONYM_MAP`，`matchRule` 时自动将关键词扩展为同义词数组再匹配。 | 大幅减少数据维护成本，玩家输入容错率提高。 |
| **阶段存档增强** | `localStorage` 保存完整对话历史、解锁标签、每轮情绪值。档案页增加"审讯复盘"入口，可回看全过程。 | 增加游戏重玩价值和成就感。 |

### 4.2 建议做（体验升级，3-7天可完成）

| 增强点 | 具体实现 | 预期效果 |
|--------|----------|----------|
| **规则优先级 + 历史门控** | rule 增加 `priority: number` 和 `requires: { unlockedTags: [], maxMood: number, minRound: number }`。引擎先过滤可触发规则，再按优先级取最高。 | 支撑"低情绪下解锁认罪台词""提及家人后再威压有特殊反应"等深度叙事。 |
| **三维心理模型（简化版）** | 将 `mood` 拆分为 `tension`（紧张度）、`defense`（防御值）、`trust`（信任度）。不同 tone 对三维的影响不同（威压：defense↑ trust↓；温和：trust↑ tension↓）。SVG立绘根据三维值微调：tension高→呼吸急促；defense高→眉头紧锁；trust高→眼神柔和。 | 玩家能感受到"每个嫌疑人有不同的软硬组合"，策略深度增加。 |
| **环境音效系统** | 使用 Web Audio API 或 `<audio>` 预加载：打字机按键音、判定错误警示音、阶段切换纸张翻动音、嫌疑人情绪崩溃时的喘息/叹息音。 | noir氛围的核心补足。注意音量控制和可关闭选项。 |
| **立绘微表情过渡** | SVG 中 mouthPath / browAngle / eyeY 不再三档跳跃，而是根据 mood 值连续插值（如 `browAngle = -8 + (mood - 70) * 0.2`）。增加眨眼动画随机触发。 | 从"表情包切换"升级为"活的肖像"。 |
| **对话输入联想/快捷语** | 在输入框下方根据当前阶段目标，显示 3-4 个常用问话语快捷按钮（如阶段1显示"谁策划的？""是你组织的吗？"）。 | 降低玩家"不知道问什么"的流失率，尤其对移动端友好。 |

### 4.3 可延后（锦上添花，7天以上或需要重构）

| 增强点 | 具体实现 | 延后原因 |
|--------|----------|----------|
| **案件编辑器（可视化）** | 独立 HTML 页面，表单化编辑案件/阶段/规则，导出 JSON 或 cases.js 格式。 | 当前3个案件手写维护成本可控，案件数量超过10个时才需要。 |
| **真正的分支结局（非线性）** | 每阶段增加多个出口，根据解锁标签组合进入不同结局（如"部分认罪""揭发同伙""精神崩溃"）。 | 需要重构 `advanceStage` 为更通用的状态机，当前线性阶段制足够支撑Demo叙事。 |
| **服务端存档 / 分享** | 将审讯记录生成可分享的哈希链接，或上传至后端。 | 纯前端约束下，分享功能可用 `location.hash + base64` 模拟，但无服务端仍无法持久化跨设备存档。 |
| **WebGL 立绘 / 3D 头像** | 用 Three.js 或 PixiJS 渲染嫌疑人头像，支持更丰富的光影和动画。 | 当前 SVG 方案已足够表达情绪，且风格统一（ noir 档案风）。3D 容易破坏现有美学。 |
| **接入外部LLM（云端）** | 通过 fetch 调用外部 API（如 OpenAI / Claude）实现真正的生成式对话。 | 一旦引入后端/LLM，项目性质从"纯前端Demo"变为"全栈应用"，部署和成本结构完全改变。建议作为"正式版"方向，而非Demo迭代。 |

## 五、当前架构瓶颈与可扩展性分析

### 5.1 瓶颈（限制天花板的地方）

1. **引擎无状态，导致叙事无法"记住"玩家**
   - `ENGINE.generateReply(stage, tone, text, mood)` 的签名中没有 `history` 参数。
   - 嫌疑人无法说出"你刚才不是还说...""第三次问我家人了"这类有记忆感的台词。
   - **影响**：案件设计只能做"独立轮次问答"，做不了"渐进式心理攻势"。

2. **单维 mood 模型过于粗糙**
   - 林婉（紧张型）和老K（冷静型）在代码层面只有 `personality` 标签用于SVG渲染，审讯逻辑上完全共享同一套 `mood` 数值体系。
   - 威压对老K应该几乎无效（冷静型防御高），但对林婉应该直接击穿心理防线（紧张型 tension 飙升）。当前引擎无法表达这种"性格-策略"的克制关系。
   - **影响**：所有嫌疑人"手感"趋同，仅靠文案区分性格。

3. **数据扁平，维护成本随案件数线性爆炸**
   - 案件2的"客服"角色需要维护一组关键词，案件4如果也有"客服"角色，同义词需要复制粘贴。
   - `responseRules` 是平面数组，没有继承/复用/模板机制。
   - **影响**：新增一个案件的数据工作量 ≈ 重写所有规则，策划无法独立维护。

4. **CSS 已趋近单文件极限**
   - `styles.css` 已超 1000 行，涵盖大厅/审讯室/结算/档案/模态/响应式。
   - **影响**：继续增加主题（如"夜间审讯室""医院审讯室"）时容易引入回归Bug。

### 5.2 可扩展性（做得好的地方）

1. **数据与引擎解耦**：新增案件只需在 `cases.js` 追加对象，无需改引擎和UI代码。
2. **阶段制结构清晰**：案件 → 阶段 → 规则的三层嵌套，天然适合扩展更多案件。
3. **纯前端无构建**：零依赖、零打包，任何改动即时生效，迭代速度极快。

## 六、新增案件：现有结构能否支撑？

### 结论：可以支撑，但有明确的天花板。

**可直接复用的部分**：
- `cases.js` 的数据结构（case / prisoner / stages / responseRules / fallback）完全够用。
- 引擎的 `generateReply` + `isGoalAchieved` + `isTrapped` 可以无改动运行新案件。
- 大厅渲染、档案页、结算页的通用逻辑无需修改。

**天花板在哪里**：
- 如果新案件需要**跨阶段状态传递**（如阶段1的选择影响阶段2的初始信任度），当前结构不支持。需要给 `stage` 增加 `carryOverState` 字段，并修改 `advanceStage` 函数。
- 如果新案件需要**动态目标**（如目标不是固定关键词，而是"累计触发3次恐惧反应"），当前 `goalKeywords` 机制不支持。
- 如果新案件嫌疑人的**性格需要全新审讯逻辑**（如"反社会人格"对温和语气极度反感、对威压反而兴奋），当前单维 `mood` 模型无法支撑。

**建议**：
- 在现有结构下，新增 **2-3 个同类型案件**（不同人设但逻辑模式相同）完全没问题。
- 若要做**第7个及以上案件**，或想做**突破性叙事**（如双嫌疑人对峙审讯、时间限制推理），必须先升级引擎（优先级 + 历史门控 + 三维心理模型）。

## 八、关键代码改动点提示

如需立即启动迭代，以下是首要改动的文件和函数：

1. **`engine.js: matchRule()`** — 增加否定检测和同义词扩展
2. **`engine.js: generateReply()`** — 增加 `history` 参数和优先级排序逻辑
3. **`app.js: state 对象`** — 将 `mood: 50` 扩展为 `{ tension: 50, defense: 50, trust: 50 }`
4. **`app.js: sendPlayerMessage()`** — 在调用 `generateReply` 时传入 `state.dialogue`
5. **`data/cases.js`** — 给 rule 增加 `priority` / `requiresUnlocked` / `negatedReply` 等可选字段
6. **`app.js: setupSpeech()`** — 在 `addMessage('prisoner', result.reply)` 后增加 `speechSynthesis.speak()`

## 九、总结

当前项目的**最大优势**是"轻量、解耦、零构建"，**最大短板**是"引擎太简单、状态太扁平"。

在纯前端无LLM的硬约束下，Demo级别的最优策略是：

> **用"更聪明的规则 + 更丰富的感知反馈"模拟智能，而不是追求真正的智能。**

优先落地 **TTS语音**、**否定检测**、**同义词扩展** 这三项，投入产出比最高；随后用 **规则优先级 + 历史门控** 打开叙事深度；最后用 **三维心理模型 + 音效** 把沉浸感推到Demo所能触及的天花板。

现有架构**足够支撑新增 2-3 个案件**，但若要持续扩展内容，引擎升级是必经之路。
